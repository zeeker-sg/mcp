- **WR-01:** `core/search.py::_one_table` augmentation lacks the

  rendered `_citation` string on search rows.
- **WR-02:** D6.1-03's `_sort=rowid` override is applied ONLY in
  `query_table`; `tools/retrieval.py::fetch` and `core/search.py::_one_table`

  `<url_col>__exact` filter (Datasette doesn't apply per-table default sort

  upstream config change away from re-triggering the same HTTP 400 trap.
- **WR-03:** `_one_table` augmentation never logs a `metadata_gap`-style
  warning when a template references a column absent from upstream
  (`available`), so config drift on the search path fails silently — the
  rendered `_citation` quietly substitutes `""` and ships a degraded
  citation without any operator signal.

Four INFO items capture style/documentation drift worth a small cleanup
pass: duplicate `import re` inside a test function, scope-boundary
docstrings that describe behavior implemented elsewhere, and one cursor
migration scenario worth noting in the release notes.

The Phase 6.1 scope boundary is respected: no new envelope keys, no new
tools, no changes to `CONTENT_POLICIES` / `CITATION_TEMPLATES` contents,
no changes to the locked PRD §12 error catalog.

## Warnings

### WR-01: Search path skips `& visible` defense-in-depth on citation augmentation

**File:** `src/mcp_zeeker/core/search.py:201-210`

**Issue:** `_one_table` computes `search_added_columns = search_placeholders - search_preview_cols` and appends every member to the upstream `_col=` list without intersecting against the visible (non-hidden) column set for `(db, table)`. By contrast, `tools/retrieval.py::query_table` lines 350-353 explicitly applies `(placeholder_cols - caller_cols) & visible` with the comment "Belt-and-suspenders: intersect with `visible` so a template referencing a hidden or unknown column cannot smuggle it into the upstream `_col=`."

Both paths share the same risk surface: if any future `CITATION_TEMPLATES` entry references a column listed in `HIDDEN_COLUMNS["*"]` or `HIDDEN_COLUMNS[<db.table>]` (for example, an operator adds `{internal_id}` to a template while `internal_id` is hidden), then on the search path:

1. The hidden column name lands in the upstream `_col=` projection.
2. Datasette returns the column value in the raw row `r`.
3. `synthesize_citation(db, table, r, retrieved_at_for_call)` reads `r["internal_id"]` via `_SafeDict.__init__` and substitutes it into the rendered `_citation` string.
4. The 9-key normalize loop at lines 249-269 emits `_citation` at the row top level.

The hidden value never appears as a top-level row key (the fixed 9-key reshape blocks that), but it DOES appear inside the `_citation` string body — which is exactly the kind of hidden-data leak the `HIDDEN_COLUMNS` invariant is meant to prevent. Because templates are operator-authored config, the immediate risk is config-correctness rather than malicious injection, but the asymmetry between the two augmentation sites is unjustified.

`available` is computed in `searchable_tables_for` at line 161 (`available = set(t.columns)`) but is not threaded through to `_one_table`. A minimal fix re-derives the visible set or threads it through `target_tables`.

**Fix:**
```python
# In _one_table, before building params:
search_template = config.CITATION_TEMPLATES.get((db, table), config.DEFAULT_CITATION_TEMPLATE)
search_placeholders = placeholder_columns(search_template)
search_preview_cols = {c for c in preview.values() if c is not None}
# Defense-in-depth: a template referencing a hidden/unknown column must
# not smuggle the column into the upstream _col= projection. Mirror the
# `& visible` guard in tools/retrieval.py::query_table (line 353).
visible_cols = await _visible_columns(db, table)
search_added_columns = (search_placeholders - search_preview_cols) & visible_cols
```

Or — preferred, avoids the extra upstream call per fan-out task — thread `available` through `target_tables` from the handler and pass it to `_one_table` as an additional argument.

### WR-02: `_sort=rowid` override is applied only in `query_table`; `fetch` and `_one_table` remain on the unfixed path

**File:** `src/mcp_zeeker/tools/retrieval.py:378-383` (fix site); `src/mcp_zeeker/tools/retrieval.py:712` (fetch — unfixed); `src/mcp_zeeker/core/search.py:205-210` (search — unfixed)

**Issue:** D6.1-03 / Finding #4 root-caused the `mlaw_news` heavy-column failure to Datasette emitting invalid SQL when `_col=` excludes the per-table default-sort column. The fix injects `_sort=rowid` when the caller does not specify a sort. However, the fix is applied ONLY in `query_table`. Two other code paths issue table-row requests against the same upstream and remain on the unfixed path:

1. **`fetch` (retrieval.py:712):** issues `params = [(url_col + "__exact", url), ("_size", "2")]` with no `_sort` param. In current production, `fetch`-supported tables (URL-keyed: judgments, enforcement_decisions, sg-gov-newsrooms.*_news, sglawwatch.*) overlap with the tables that exhibited Finding #4 (`sg-gov-newsrooms.*_news` all have a default-sort directive). The `__exact` filter on the URL column likely masks the trap because Datasette's per-table default sort is applied AFTER filter evaluation and `__exact` produces a unique row, so the SQL plan likely degenerates harmlessly. But the masking is incidental, not principled. A future upstream metadata change (e.g., changing default sort to a heavy column on a URL-keyed table) could re-trigger Finding #4 on `fetch`.

2. **`_one_table` (search.py:205-210):** issues `params = [("_search", escaped), ("_size", str(per_table_limit)), *_col entries]` with no `_sort` param. Search relies on `_search=` which provides its own FTS ranking, so Datasette's per-table default sort is typically overridden by the FTS rank. Empirically safe today, but again the safety is incidental — FTS-on + per-table-default-sort interaction is upstream behavior the connector does not control.

D6.1-03's stated rationale ("uniform override that works on every table") applies equally to both paths. Recommend mirroring the fix into both for consistency and resilience against upstream metadata drift. The cost is two extra URL params on every search-task / fetch dispatch.

**Fix:**
```python
# In fetch (retrieval.py:712):
params: list[tuple[str, str]] = [
    (url_col + "__exact", url),
    ("_size", "2"),
    ("_sort", "rowid"),  # D6.1-03 parity: see query_table line 383
]

# In _one_table (search.py:205-210):
params: list[tuple[str, str]] = [
    ("_search", escaped),
    ("_size", str(per_table_limit)),
    ("_sort", "rowid"),  # D6.1-03 parity — defensive against upstream metadata drift
    *[("_col", c) for c in preview.values() if c is not None],
    *[("_col", c) for c in sorted(search_added_columns)],
]
```

### WR-03: `_one_table` augmentation silently degrades when template references a column upstream doesn't return

**File:** `src/mcp_zeeker/core/search.py:201-274`

**Issue:** `_one_table` augments `_col=` with every placeholder column from the citation template (line 209) but never verifies that upstream actually returned those columns in the row dict `r`. If a template references a column that exists in upstream `t.columns` (so the `& visible` defense in WR-01 would pass) but is absent from the actual row payload (e.g., a sparse-field row), `_SafeDict.__init__` substitutes `""` for the missing field and `_citation` renders with empty-placeholder punctuation (the `"  (, ) — "` form Finding #2 captured at the envelope boundary).

No `metadata_gap`-style INFO log fires for this case. Compare with `MetadataCache.get_column_description` (lines 153-166 of metadata_cache.py) which emits `metadata_gap` when an expected field is absent. The search-path augmentation has no equivalent observability seam.

This is the operator-visibility analog of WR-01 — both protect the citation-template contract on the search path, and both currently rely on inspection rather than runtime signals. Recommend a light INFO log when `set(search_placeholders) - set(r.keys())` is non-empty for the first row of a per-table response.

**Fix:**
```python
# Inside the per-row loop, BEFORE synthesize_citation:
if raw_rows:
    missing_placeholders = search_placeholders - set(raw_rows[0].keys())
    if missing_placeholders:
        log.info(
            "citation_placeholder_gap",
            database=db,
            table=table,
            missing=sorted(missing_placeholders),
        )
```

The same gap exists on `query_table` (retrieval.py:551) — `synthesize_citation` is called on `upstream_row` without verifying that the augmented placeholder columns landed. Apply the same observability hook there for symmetry.

### IN-02: Cursor migration across the `_sort=rowid` rollout

**File:** `src/mcp_zeeker/tools/retrieval.py:378-453`

**Issue:** A cursor encoded by a pre-D6.1-03 server (no `_sort=rowid` default) carries an upstream `next` token computed under whatever per-table default sort Datasette applied. If a client persists that cursor across a server upgrade and re-issues it post-fix, the server's `canonical_shape_str` matches (same `sort=None`), so `decode_cursor` succeeds, but the new request issues `_sort=rowid` to upstream — Datasette will interpret the persisted `_next` token under rowid ordering instead of the previous default sort, potentially producing duplicate or skipped rows on the boundary page.

This is a transient, one-shot migration concern: once any client has re-issued a fresh `query_table` call post-fix, all subsequent cursors are consistent. It is not a correctness regression in steady state. Worth mentioning in the release notes / phase summary for the operator.

**Fix:** Add a release-note bullet documenting the transient cursor inconsistency window. Optional belt-and-suspenders: include a sort-mode token in the qhash payload (e.g., `"rowid"` vs `"caller-specified"`) so pre-fix cursors deterministically fail with `invalid_cursor` instead of silently re-interpreting against a different ordering. This is intrusive and probably overkill for a Phase 6.1 fix-up — flag for the operator's discretion.

### IN-03: `placeholder_columns` accepts (but does not document) format-spec / index syntax

**File:** `src/mcp_zeeker/core/citation.py:72-96`

**Issue:** `placeholder_columns` uses `string.Formatter().parse(template)` and treats `field_name` verbatim as a column name. For non-trivial templates the field name can include indexing or attribute access:

- `{x[0]}` → `field_name == "x[0]"` (whole bracket expression returned by parse)
- `{a.b}` → `field_name == "a.b"`
- `{a:>10}` → `field_name == "a"` (format spec is stripped — correct)

The citation.py docstring at line 19-22 explicitly forbids `{x[0]}` and `{x.y}` templates, but nothing in code enforces that constraint, and `placeholder_columns` would happily return `{"x[0]"}` or `{"a.b"}` as "column names." On the augmentation path this is benign — the intersection `& visible` (in `query_table`) or the upstream `_col=` payload (in `_one_table` per WR-01) rejects/filters bogus names. On the `format_map` substitution path in `synthesize_citation`, `_SafeDict.__getitem__` returns `""` for any non-bare key.

No fix strictly required. Worth a one-line note in `placeholder_columns` docstring clarifying that non-bare placeholders are not stripped — they're returned verbatim and benign because the consuming sites filter or default them. The strongest mitigation is an AST/regex lint in `tests/test_citation_synthesis.py` asserting `CITATION_TEMPLATES` values match `^[^{}]*({[a-zA-Z_][a-zA-Z0-9_]*}[^{}]*)*$` so the operator-config contract stays bare-key only.

**Fix:** Add a docstring clarification:
```python
def placeholder_columns(template: str) -> set[str]:
    """Extract upstream column names referenced as `{name}` placeholders.

    Returns `field_name` verbatim from `string.Formatter().parse`. Bare
    placeholders (`{x}`) return as `"x"`. Index / attribute placeholders
    (`{x[0]}`, `{x.y}`) — forbidden by the citation.py docstring — return
    as `"x[0]"` / `"x.y"`; the consuming sites filter these out naturally
    because they don't match any upstream column. The bare-key constraint
    is enforced by inspection of CITATION_TEMPLATES; no runtime check.
    """
```

### IN-04: `_one_table` license-fallback comment misrefers to a sibling helper

**File:** `src/mcp_zeeker/core/search.py:238-243`

**Issue:** The comment at lines 238-239 states "Mirrors the `_license_pair` helper in core/envelope.py (Plan 06-02 Task 1)." The helper in `core/envelope.py` at lines 48-51 (the `_license_pair` function) is the same shape — try `MetadataCache.current().license_for_sync(...)`, except `RuntimeError` → fall back to `config.LICENSES.get(database, ("", ""))`. So the inline duplicate is correctly mirroring the contract, but the search-path code DOESN'T import or use `_license_pair` — it reimplements the same 3-line block inline. Future drift between the two sites is now a maintenance concern.

Not a correctness bug. Worth either:
- Extracting `_license_pair` into a publicly importable helper (e.g., move from `envelope.py` to `metadata_cache.py` as `MetadataCache.license_pair(database)` static method that wraps the RuntimeError-catch) and call it from `_one_table`, `query_table` (retrieval.py:533-536), and `_license_pair` in envelope.py.
- Or leave the three sites as triplet inline blocks but add a comment noting that all three sites must drift together.

**Fix:** Promote the fallback to a single helper:
```python
# In core/metadata_cache.py:
@staticmethod
def license_pair_or_config(database: str) -> tuple[str, str]:
    """Return (license, license_url) — bound cache wins, else config.LICENSES,
    else ("", ""). Single source of truth for the cache-unbound fallback path
    used by Envelope factories, query_table _policy synthesis, and search.py
    per-row license attachment."""
    try:
        return MetadataCache.current().license_for_sync(database)
    except RuntimeError:
        from mcp_zeeker import config
        return config.LICENSES.get(database, ("", ""))
```

Then `core/search.py:240-243`, `tools/retrieval.py:533-536`, and `core/envelope.py:48-51` all call `MetadataCache.license_pair_or_config(database)`.
